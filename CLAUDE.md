# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

PR Review Agent: receives GitHub `pull_request` webhooks (filtered to `opened`/`synchronize` only), validates and queues them for async analysis via Celery/Redis, checks out the PR branch, and runs a LangGraph **multi-agent review panel** (Groq-hosted `llama-3.3-70b-versatile`) — a Security Warden and a Performance & Logic Architect, each using MCP tools to read only the PR's changed files (plus their local dependencies) and statically review them within their own scope, followed by a Synthesizer that merges both into one structured review. The review is posted back to GitHub as a formal PR review + commit status, with concrete fixes rendered as clickable inline `suggestion` blocks. All panel state is checkpointed to Redis, so a worker crash mid-review doesn't lose progress.

The pipeline is hardened end to end against its two real operational risks: exhausting Groq's tight free-tier quota, and running untrusted PR-supplied code. Webhook redeliveries are deduplicated, draft PRs and budget-exhausted days are skipped before any Groq call, every Groq call's token usage/cost is recorded to Redis, retries are scoped to genuinely transient provider errors, a commit status always resolves (never stuck on `pending`) even after retries exhaust, and the MCP server that runs the PR's own test suite executes it with a stripped environment and (where possible) dropped privileges.

The project has three top-level pieces:
- `app/` — FastAPI webhook receiver + Celery worker + LangGraph review agent
- `mcp_servers/` — standalone stdio MCP servers exposing repo actions to the agent; not imported by `app/`
- `evaluation/` — golden dataset + scripts for measuring review-panel quality against known, deliberately-injected issues (see Evaluation harness below). `run_reviews.py` legitimately imports `app.agent` to drive real reviews; `judge_results.py` and the dataset itself are deliberately independent of `app/`'s config/secrets

## Commands

```bash
# Setup
python -m venv .venv
source .venv/Scripts/activate        # Windows Git Bash; use .venv\Scripts\Activate.ps1 in PowerShell
pip install -r requirements.txt

# Run the FastAPI app
uvicorn app.main:app --port 8000

# Run the Celery worker (requires Redis reachable at CELERY_BROKER_URL)
celery -A app.worker.celery_app worker --loglevel=info -P solo   # -P solo is required on Windows; prefork isn't supported

# Tests -- fast/mocked suite, no live Redis/Groq/GitHub calls (default; see pytest.ini)
pytest tests/ -v
pytest tests/test_webhook.py::test_webhook_accepts_valid_pull_request_event -v   # single test

# Live integration test -- real Groq + real Redis + real MCP subprocesses; costs real
# tokens (~19k observed for one small fixture, ~$0.01) -- run deliberately, not routinely
pytest tests/test_integration.py -m integration -v

# Run an MCP server directly (stdio transport)
python mcp_servers/code_server.py
python mcp_servers/tester_server.py

# Fire a signed mock webhook at a locally running app+worker, against a real PR
python simulate_pr.py --pr-number <n> [--full-name owner/repo]

# Forward *real* GitHub webhook events (gh-webhook extension) instead of simulating one
gh webhook forward --repo=<owner/repo> --events=pull_request --url="http://localhost:8000/webhook/github" --secret="<value matching GITHUB_WEBHOOK_SECRET>"
```

`github_webhook_secret`, `groq_api_key`, and `github_api_token` have no defaults and are required for `app/config.py`'s `Settings` to construct — set them via `.env` (copy `.env.example`) or the environment before importing `app.main`/`app.worker`/`app.tasks`, otherwise imports fail at startup. `git_workspace_root` (default `app/workspace`) is where PR checkouts live; `groq_model` defaults to `llama-3.3-70b-versatile`. `Settings.model_config` sets `extra="ignore"` so unrelated keys already present in a shared `.env` don't break construction — e.g. `GEMINI_API_KEY`/`GEMINI_MODEL`, which only `evaluation/judge_results.py` reads (directly from the environment/`.env`, not through `Settings` — that script needs none of the Groq/GitHub secrets). Confirmed the hard way: adding `GEMINI_API_KEY` to `.env` without `extra="ignore"` breaks `Settings()` construction, and therefore the entire app, since every entry point calls `get_settings()` at import time — `pydantic_core.ValidationError: Extra inputs are not permitted`.

There's a Redis instance available locally in a Docker container named `agent-redis` (port 6379) for manual end-to-end testing of the worker — check `docker ps` before spinning up a new one. It also backs LangGraph's Redis checkpointer (see below), which requires the RediSearch + RedisJSON modules (`FT.CREATE`-style index commands) — plain `redis:alpine` lacks these on older Redis, but Redis 8's `alpine` image bundles them natively, confirmed via `MODULE LIST` against this exact container. If you ever swap to a genuinely module-less Redis, `AsyncRedisSaver.asetup()` will fail at startup.

Groq's free tier is tight for this panel (12k TPM / 100k TPD as observed): a single review can burn 8-12k tokens across two specialists + a synthesizer call, and retrying a partially-completed thread makes it worse, not better, since the resumed specialist's checkpointed conversation grows on each retry instead of shrinking. The two specialists run **sequentially** (not concurrently) specifically to keep peak per-minute usage down against this limit — see `_build_panel_graph` in `app/agent.py`.

`get_settings()` is `@lru_cache`'d, so a running `celery` worker process keeps whatever `GROQ_API_KEY`/other `.env` values were in effect when it first called `get_settings()` — editing `.env` after the worker has started does **not** take effect until the worker process itself is restarted. Confirmed directly: swapping `GROQ_API_KEY` in `.env` and re-firing the same webhook against an already-running worker kept hitting the *old* key's exhausted quota until the worker was stopped and restarted.

## Architecture

### Webhook flow (`app/`)
`app/main.py`'s `POST /webhook/github` is the single entry point:
1. Reads the raw request body (needed both for signature verification and JSON parsing — don't call `request.json()`, it consumes the body).
2. Verifies `X-Hub-Signature-256` via HMAC-SHA256 against `settings.github_webhook_secret` using `hmac.compare_digest`. Missing/invalid → `401`.
3. Checks `X-GitHub-Event`; anything other than `"pull_request"` short-circuits to `200` without touching Celery (GitHub sends many event types, e.g. `ping`, that must not error).
4. Validates the body against `app/schemas.py`'s `PullRequestEvent` (nested `Repository` / `PullRequest` / `Head` models). `PullRequest.modified_files`/`.added_files` default to `[]`, since a genuine GitHub webhook never carries a file list (only `simulate_pr.py`'s payload populates them client-side). Validation failure → `422`.
5. Checks `event.action` against `_PROCESSABLE_ACTIONS = {"opened", "synchronize"}`; anything else (`review_requested`, label changes, `closed`, `reopened`, etc.) is logged and short-circuits to `200` without touching Celery — a `pull_request` webhook fires for far more than just "code changed," and only these two actions represent an actual diff worth reviewing.
6. Acquires a Redis dedup lock (`lock:review:{repo}:{pr}:{sha}`, `SET NX EX 900`, via `_get_redis_client()`) before dispatching — GitHub redelivers a webhook whenever this endpoint is slow or replies non-2xx, and a redelivery would otherwise queue a second full review (burning another review's worth of the tight Groq budget) for a commit already being analyzed. A held lock short-circuits to `200` without touching Celery; the 15-minute TTL self-expires rather than needing an explicit release.
7. Dispatches `process_pr_review_task.delay(event.model_dump())` and returns `202` immediately — the endpoint never blocks on the actual analysis.

`app/config.py`'s `Settings` (pydantic-settings) is the only source of config, loaded via the cached `get_settings()`. `app/worker.py` instantiates the shared `celery_app` from these settings.

`app/tasks.py`'s `process_pr_review_task`:
1. Skips draft PRs immediately (`pull_request["draft"]`), before any Groq/checkout work — no commit status is posted here, since nothing's been marked `pending` yet at this point.
2. Checks `app/telemetry.py`'s `check_budget_ok(settings.groq_model)` before touching checkout or the agent. On exhaustion, posts commit status `"success"` (deliberately not `"failure"` — GitHub's classic Commit Status API has no true "neutral" state, and running out of the shared daily Groq quota is an ops constraint, not a defect in this PR, so it shouldn't block a merge gated on this check the way a real review failure should) with "Review deferred: daily Groq token budget reached", and returns without touching checkout or the agent.
3. If the webhook's `modified_files`/`added_files` are both empty (a genuine GitHub webhook, as opposed to `simulate_pr.py`'s), calls `app/github_client.py`'s `list_changed_files` to fetch them via the GitHub API and mutates them into the `pull_request` dict before proceeding — otherwise `target_files` would stay empty, and since the agent isn't allowed to list the whole repo (see below), it would have nothing to review.
4. Posts a `pending` commit status, then calls `app/git_ops.py`'s `ensure_repo_checkout` (shallow, single-branch clone/fetch — see below) to get a local working copy.
5. Runs `app/agent.py`'s `run_pr_review_agent` against that checkout and returns its result.

**Retries and failure handling.** The task decorator sets `autoretry_for=(RateLimitError, APIConnectionError, APITimeoutError)` — Groq's own transient-failure exception types, imported directly from the `groq` package — with `max_retries=3, retry_backoff=True, retry_jitter=True`. Deliberately narrow, not `autoretry_for=(Exception,)`: a local bug (e.g. a `KeyError` on malformed `event_data`) should fail on the first attempt and show up in tracking immediately, not get masked behind a 3x exponential backoff loop that can't fix it. For this to retry anything at all, `app/agent.py`'s `run_pr_review_agent` has a matching carve-out in its own broad `except Exception`: those same three exception types are re-raised past that handler instead of being swallowed into an `"error"` status dict — otherwise nothing would ever reach the task decorator's `autoretry_for` in the first place, since the agent's catch-all runs first.

A custom `_CommitStatusOnFailureTask` (set as the task's `base=`) implements `on_failure`, which Celery calls exactly once for a task's *genuinely terminal* failure — never for an individual retry attempt, since a retry raises an internal `Retry` control-flow exception instead (`celery/app/autoretry.py`'s `add_autoretry_behaviour` wraps `task.run` itself to do this). This is what stops a PR's commit status from being stuck on `pending` forever once all 3 retries (or a non-retryable crash) exhaust: `on_failure` posts a final `"failure"` status ("Review aborted: execution failure or provider exhaustion") exactly once, without flickering `pending`/`failure` across intermediate retries the way notifying on every exception inside the task body would.

### Token telemetry and budget guard (`app/telemetry.py`)
Every Groq call the panel makes is instrumented. `_run_specialist` sums `usage_metadata` (`input_tokens`/`output_tokens` — LangChain's standardized field names, confirmed against `langchain_groq`'s own `_create_usage_metadata`) across every `AIMessage` a specialist's ReAct loop produces via `_sum_usage_metadata`. `_synthesize` switched from a plain `with_structured_output(ReviewOutput)` call to `include_raw=True` specifically to reach the raw `AIMessage`'s usage data, since the parsed `ReviewOutput` alone doesn't carry it — a parsing failure still surfaces as `parsed=None`, handled by the pre-existing "`final_review` is `None`" branch downstream, not a new failure mode.

`record_usage(model, prompt_tokens, completion_tokens)` atomically pipelines `INCRBY`/`INCRBYFLOAT` + `EXPIRE` (7 days) across `usage:groq:{prompt_tokens,completion_tokens,cost}:{UTC date}` in Redis, and fails open (logs and swallows) on a Redis error rather than breaking the review it's instrumenting. **Don't cache the async Redis client at module level** — `_new_redis_client()` deliberately opens a fresh `redis.asyncio.Redis` connection on every call. A cached one is bound to whichever asyncio event loop was running when it first connected, and `app/tasks.py` runs each review via its own fresh `asyncio.run(...)` (a new event loop every time) — reproduced directly: caching the client made the second of two sequential `asyncio.run(record_usage(...))` calls against a real Redis fail with "Event loop is closed" and silently drop its update.

`calculate_cost(model, prompt_tokens, completion_tokens)` looks up USD-per-1M-token pricing from a small hardcoded dict (currently only `llama-3.3-70b-versatile`: $0.59 prompt / $0.79 completion, from Groq's published pricing as of writing — re-verify against their live pricing page before trusting this for anything billing-sensitive) and returns $0.00 with a warning log for any unrecognized model.

`check_budget_ok(model, safe_limit=90_000)` gates `process_pr_review_task` before any checkout/agent work: sums today's `prompt_tokens` + `completion_tokens` from Redis and returns `False` if at/over `safe_limit`. Deliberately **synchronous** — a plain `redis.Redis` client, cached via `lru_cache` — unlike `record_usage`'s async client: this runs at the very top of the task before any `asyncio.run(...)` call exists, so there's no event loop to bind to and caching is safe here (the distinction that broke `record_usage` above doesn't apply to a sync client). Fails open (returns `True`) if Redis is unreachable. Note the usage keys aren't dimensioned by model — every Groq call currently draws from one shared daily budget regardless of which specialist made it, so `model` is accepted for log messages and symmetry with `calculate_cost`/`record_usage`, not to select a different key.

### Review agent: multi-agent panel (`app/agent.py`)
Instead of one generic reviewer, a LangGraph `StateGraph` (`PanelState`) runs a fixed pipeline: **Security Warden** → **Performance & Logic Architect** → **Synthesizer**, built per-invocation by `_build_panel_graph`. The two specialists run sequentially, not concurrently — see the Groq rate-limit note above; the graph would work identically with `START` fanning out to both (they don't depend on each other's output) if that constraint ever goes away.

- Each specialist is its own `create_react_agent` (Groq `llama-3.3-70b-versatile`, `temperature=0.1`), scoped via prompt to one concern (`SECURITY_WARDEN_PROMPT`: data leaks/dependency vulns/OWASP/injection; `PERFORMANCE_ARCHITECT_PROMPT`: algorithmic correctness/edge cases/Big-O/control flow, including silent failure modes — off-by-one data loss, races, stale/incorrectly-keyed caches — treated as just as real as ones that throw) and restricted to `_SPECIALIST_TOOL_NAMES = {fetch_file_contents, scan_local_dependencies, run_validation_suite}`. **Neither has `apply_code_patch`** — deliberately: they're read-only reviewers, since arbitrating conflicting patches from two independent agents was out of scope. A concrete fix is instead described as a `SUGGESTION file=<path> line=<n>: <code> || <reason>` marker embedded in the specialist's own findings text (per `_PANEL_TOOL_DISCIPLINE`). `fetch_file_contents` (see MCP servers below) returns each line prefixed with its exact 1-indexed number specifically so a specialist can copy a line number rather than count it from raw text — added after a real run produced a `SUGGESTION` with an off-by-one line number, then extrapolated a second suggestion from that wrong anchor to a line past the end of the file, which GitHub's review-comment API rejected with `422: Line could not be resolved`. `_PANEL_TOOL_DISCIPLINE` also tells every specialist to verify a claimed runtime behavior against actual language/library semantics before asserting it in findings, preferring "None found." over an unverified guess — added after the Performance & Logic Architect covered a genuine miss with a confidently-stated but factually wrong claim (e.g. that Python list slicing raises `IndexError` on an out-of-range end index, or that `dict.get(key, default)` raises `KeyError`) instead of just reporting nothing for that scope.
- Each specialist writes to its own disjoint `PanelState` keys (`security_findings`/`performance_findings`, etc.) rather than a shared `messages` channel — required for LangGraph to allow two nodes to update state in the same superstep without an `InvalidUpdateError`, and kept even now that they're sequential so re-parallelizing later is a one-line edge change, not a state-schema rewrite.
- Each specialist gets its own structured extraction pass (`response_format=(SPECIALIST_STRUCTURED_PROMPT, SpecialistFindings)`), same reasoning as the old single-agent design: the ReAct loop's injected `prompt=` isn't visible to that separate pass.
- The **Synthesizer** (`_synthesize`) is a plain `ChatGroq.with_structured_output(ReviewOutput)` call — no tools, no ReAct loop — that dedupes overlap between the two reports, resolves conflicts (favoring security/correctness over raw performance), routes each point into the right `ReviewOutput` section, and extracts `SUGGESTION` markers verbatim into `inline_suggestions`.
- **Grounding gotcha (found via live testing, not hypothetical):** the Synthesizer only ever sees the specialists' distilled findings text, never the original tool-call transcript. Early on, it would paraphrase issues without naming the file (e.g. "the function has an O(n²) loop" instead of "in `inefficient.py`..."), which made the post-hoc `_is_grounded()` check fail even though the review was completely legitimate. Fixed by requiring both `SPECIALIST_STRUCTURED_PROMPT` and `SYNTHESIZER_PROMPT` to explicitly preserve the exact file name for every issue carried forward — if you touch either prompt, keep that requirement or grounding will start failing silently again.
- **Section-routing gotcha (found via the 15-fixture golden dataset in `evaluation/`, not hypothetical):** the Synthesizer would sometimes route a structural/maintainability finding (e.g. duplicated code) into `performance_observations` — because the reporting specialist's own wording sounded performance-flavored ("redundant calculations that can be simplified") — or drop it from every structured section entirely while still describing it in `summary`. Fixed by making `SYNTHESIZER_PROMPT` and both the `performance_observations`/`architectural_suggestions` field descriptions on `ReviewOutput` explicit that routing follows what an issue *is*, not which specialist reported it or how they phrased it, plus a rule that nothing may appear only in `summary`. Verified directly: replaying the exact real specialist findings that previously mis-routed now correctly land in `architectural_suggestions`, and both previously-missed performance fixtures were caught cleanly (no fabricated claims) on the first re-run after the `PERFORMANCE_ARCHITECT_PROMPT` change above.
- `_is_grounded()` itself is unchanged in spirit from the single-agent design: the final formatted review must mention at least one file basename from the *union* of both specialists' `fetched_files`. A grounding failure, a missing `final_review`, or either specialist's circuit breaker tripping (`security_circuit_broken`/`performance_circuit_broken`, same `MAX_CONSECUTIVE_TOOL_ERROR_TURNS` logic as before, per-specialist) results in `status: "error"` and a `failure` commit status — fail-closed, never post a partial/degraded review.
- `ReviewAgentState` (per-specialist, custom `AgentState` subclass) must still declare `structured_response` and `target_files` itself, same reason as before: a custom `state_schema` disables `create_react_agent`'s own `AgentStateWithStructuredResponse` auto-fields.

**Redis checkpointing** threads through the whole panel lifecycle. `_thread_id_for(repository, pr_number, sha)` → `pr:{repo}:{pr_number}:{sha}` is the *base* thread id:
- The panel graph itself is checkpointed under the base id.
- Each specialist gets its own sub-thread, `{base_thread_id}:security` / `{base_thread_id}:performance`, on the same `AsyncRedisSaver` connection (opened once per `run_pr_review_agent` call via `async with AsyncRedisSaver.from_conn_string(settings.redis_url) as checkpointer`, `asetup()` called every time — idempotent, cheap relative to everything else in a review).
- The thread id is scoped to the **head sha**, not just repo+PR: a later `synchronize` event (new commits) gets a completely fresh, isolated thread rather than resuming the previous commit's tool-call history, since that history reflects code that no longer exists and would pollute the new review. A retried/crashed task for the *same* sha does correctly resume.

### Git checkout (`app/git_ops.py`)
`ensure_repo_checkout` clones shallow and single-branch (`git clone --depth=1 --single-branch --branch <branch>`) on first use per repo, then reuses that checkout for later PRs via `git fetch --depth=1 origin <branch>` + `checkout --force -B <branch> FETCH_HEAD` + `reset --hard` + `clean -fd` — this works fine even for a different branch than the one originally cloned, since checkout always targets `FETCH_HEAD` directly rather than a tracked `origin/<branch>` ref. The `--force` on checkout and the trailing `clean -fd` aren't defensive filler: without `--force`, a previous review's leftover uncommitted change to a tracked file makes plain `checkout -B` abort with "local changes would be overwritten" *before* `reset --hard` ever runs — reproduced directly against a real stale checkout, not theoretical. `clean -fd` (no `-x`) removes untracked leftovers the same way without touching anything the target repo's own `.gitignore` excludes.

### GitHub API calls (`app/github_client.py`)
`set_commit_status`, `post_review` (posts a formal PR review via `create_review`; pass `comments` — a list of `{"path", "line", "side", "body"}` dicts — to attach inline suggestion comments anchored to a specific commit, resolved via a `Commit` lookup since GitHub requires line comments tied to a `commit_id`; omit `comments` for a plain summary-only review), and `list_changed_files` (fetches real `(modified_files, added_files)` via the PR files API — used both by `app/tasks.py`'s fallback and by `simulate_pr.py`). All raise `GitHubNotifyError` on API failure; callers log-and-continue rather than let a notification failure crash the review.

Concrete fixes reach GitHub as clickable suggestions: `app/agent.py`'s `_build_inline_comments` converts each `InlineSuggestion` (from `ReviewOutput.inline_suggestions`) into a `` ```suggestion ``` ``-wrapped comment body, but drops any suggestion whose file isn't in the grounding-checked `fetched_files` set first — same fabrication guard as `_is_grounded`, applied per-suggestion, since the model could otherwise invent a plausible-looking file/line pair that was never actually read.

### Testing
`pytest.ini` sets `addopts = -m "not integration"`, so a bare `pytest`/`pytest tests/` runs only the fast, fully-mocked suite (no live Redis/Groq/GitHub calls). Every test file that imports `app.main`/`app.worker`/`app.tasks` sets `GITHUB_WEBHOOK_SECRET`/`GROQ_API_KEY`/`GITHUB_API_TOKEN` via `os.environ.setdefault(...)` **before** that import, since `Settings()` is constructed at import time (in `app/worker.py`) and has no defaults for those fields — any new test touching those modules must follow the same pattern.

- `tests/test_webhook.py` mocks `app.main.process_pr_review_task.delay` and `app.main._get_redis_client` (the dedup lock's client) directly, and computes real HMAC signatures per-request via a `_sign()` helper.
- `tests/test_tasks.py` covers `process_pr_review_task`'s control flow — draft skip, budget-exhausted skip, the normal path, and the `on_failure` hook (both its own logic in isolation and its real wiring through Celery's tracer via `process_pr_review_task.apply(...)`) — with `ensure_repo_checkout`/`run_pr_review_agent`/`set_commit_status`/`check_budget_ok` mocked at the `app.tasks.*` names they're imported under.
- `tests/test_agent_pipeline.py` covers `mcp_servers/code_server.py`'s `scan_local_dependencies` (real temp files via `tmp_path` rather than mocking `builtins.open` — the tool's `is_dir`/`resolve`/`is_relative_to` guards hit the real filesystem regardless of an `open` mock, so real throwaway files give equivalent isolation more robustly), `app/agent.py`'s grounding/formatting guardrails (`_fetched_file_names`, `_is_grounded`, `_format_review`, `_build_inline_comments`), and `_build_panel_graph`'s topology — patching `_run_specialist`/`_synthesize` (the two actual `ChatGroq` call sites) rather than `ChatGroq` itself, run against a real compiled graph and an in-memory `MemorySaver` checkpointer, to prove the real `StateGraph` wiring/state reducers work without needing a real LLM or Redis.
- `tests/test_telemetry.py` covers cost calculation, `record_usage`'s Redis pipeline (mocked `redis.asyncio` client/pipeline, including a regression test for the event-loop-caching bug described above), and `check_budget_ok`'s under/over-limit/fail-open behavior (mocked sync client).
- `tests/test_integration.py` is the one file that isn't mocked at all — see below.

`simulate_pr.py` exercises the full pipeline end-to-end against a real PR: looks up branch/sha/clone_url/changed-files via the GitHub API (through `app/github_client.py`'s `list_changed_files`), builds a signed webhook payload, and POSTs it to a locally running `uvicorn`+`celery` pair. For testing against genuinely GitHub-originated webhooks instead of a simulated payload, use `gh webhook forward` (the `gh-webhook` CLI extension) pointed at the same local endpoint with a matching `--secret`.

**Live integration test** (`tests/test_integration.py`, marked `integration`, excluded from the default run by `pytest.ini`): the only test that makes a real `ChatGroq` call, spawns real `code_server.py`/`tester_server.py` MCP subprocesses, and uses a real Redis (checkpointer + telemetry) — run explicitly with `pytest tests/test_integration.py -m integration -v`. Every other test file mocks `ChatGroq` entirely, so nothing else in the suite would catch a real prompt/tool-schema/structured-output contract break (a model swap, a `langchain_groq` version bump, a prompt edit). Skips cleanly (not an error) if there's no real `GROQ_API_KEY` or Redis isn't reachable; the key check goes through `get_settings()` rather than raw `os.environ`, since a real key living only in `.env` (pydantic-settings' source order is env vars > `.env` > field defaults, per `BaseSettings.settings_customise_sources`) would otherwise be invisible to a naive env check. **Not free**: one run costs real Groq tokens (~19k observed for a single small fixture, ~$0.01) — run it deliberately (pre-merge, on-demand), not on every commit; at the default `check_budget_ok` `safe_limit=90_000`, two runs alone would burn over 40% of a day's budget.

### MCP servers (`mcp_servers/`)
Two independent `FastMCP` apps, each launched as its own stdio process — they are not imported by `app/` and don't share code with each other by design (duplicated helpers are intentional, not an oversight).

- `code_server.py` (`PR-Code-Server`): `list_repo_files`, `fetch_file_contents`, `scan_local_dependencies`, `apply_code_patch`. All file-path tools route through `_resolve_within_repo(repo_path, file_path)`, which resolves the target path and rejects anything that escapes `repo_path` (via `Path.is_relative_to`) — required because `file_path`/`proposed_code` ultimately originate from LLM output or PR content and must be treated as untrusted.
  - `list_repo_files` is excluded from the review agent's toolset specifically (see above) but is still a normal tool on this server otherwise.
  - `fetch_file_contents` detects encoding via `charset_normalizer` rather than assuming UTF-8 — a hardcoded UTF-8 decode silently "succeeds" on a UTF-16 file, producing null-byte-interleaved garbage instead of an error. It also prefixes every line with its exact 1-indexed line number (`"N | code"`) before returning — added after the panel emitted an inaccurate `SUGGESTION` line number for a file with two near-identical functions; without numbering, a specialist has to count lines from raw unnumbered text, which is unreliable (see the Review agent section above).
  - `scan_local_dependencies` parses a file with `ast` and resolves its local (same-project) imports — both absolute (`import a.b`) and relative (`from . import x`, `from ..x import y`) — to actual file paths under `repo_path`, silently skipping anything that doesn't resolve to a real project file (stdlib/third-party imports).
  - Tools return `"Error: ..."` strings rather than raising, since MCP tool results are plain strings.
- `tester_server.py` (`PR-Tester-Server`): `run_validation_suite` shells out to `pytest` via `subprocess.run(cwd=repo_path, timeout=120)` in a worker thread (`asyncio.to_thread`, not `asyncio.create_subprocess_exec` — nesting the latter in this server's own stdio event loop deadlocks on Windows). Critically, that `subprocess.run` also passes `stdin=subprocess.DEVNULL`: without it, pytest inherits this server's own stdin (the live MCP JSON-RPC pipe) and hangs for the full 120s regardless of what the target repo's tests actually contain — confirmed by direct reproduction, not theoretical. It also appends `--timeout=15` (pytest-timeout) as a second, per-test safety net, but only after probing the resolved pytest binary's own `--help` output to confirm the plugin is actually installed there — a different repo's own `.venv` pytest might not have it, and blindly passing an unsupported flag would break every validation run for that repo.
  - **This runs untrusted PR code by design** — pytest collection executes module-level code in every collected file/`conftest.py`, not just inside `def test_*` functions — so this subprocess is sandboxed on two independent axes. `_sanitized_subprocess_env()` passes an allowlisted env (`PATH`, plus the handful of vars Windows' own interpreter/tempfile handling needs to start at all: `SYSTEMROOT`, `TEMP`, `TMP`, `PATHEXT`, `COMSPEC` — empirically confirmed sufficient for a real pytest run) instead of inheriting this process's full environment, so a malicious PR's test code can't read `GROQ_API_KEY`/`GITHUB_API_TOKEN`/`REDIS_URL`/etc. via `os.environ` and echo them into stdout/stderr, which would otherwise flow straight into the LLM's context and potentially into a public GitHub PR comment (verified directly: a simulated exfiltration attempt via `os.environ.get(...)` inside a test file returned `None` under the sandboxed env, confirmed by forcing the test to fail so pytest surfaced the captured value). `_sandbox_subprocess_kwargs()` additionally passes `user="nobody"` via `subprocess.run`'s POSIX-only `user=` kwarg, but only when actually running as root and `nobody` exists on the system — Windows has no equivalent at all, and a worker already running unprivileged can't drop further, so both cases fail open to no extra kwargs rather than erroring. This is defense-in-depth on top of the env stripping, not a full jail: dropping uid alone doesn't isolate the filesystem or network, and `nobody` can only exec pytest at all if `git_workspace_root` checkouts are readable by that user — a deployment-time permission to configure separately, not something this code guarantees. A `PermissionError` from that mismatch is already caught and reported as a clean `"Error: ..."` string rather than crashing.

When adding tools to either server, preserve this convention: validate inputs and return a descriptive `"Error: ..."` string on failure instead of letting exceptions propagate to the MCP client.

### Evaluation harness (`evaluation/`)
A golden dataset plus three independently runnable scripts for measuring review-panel *quality* against known, deliberately-injected issues — separate from `tests/`, which only verifies plumbing correctness.

`evaluation/golden_dataset/` holds 15 synthetic PR fixtures, 5 each under `security/`, `performance/`, `structural/` — one deliberately injected issue per fixture (SQL injection, an off-by-one pagination bug, duplicated functions, etc.). Each is a self-contained mini-repo: `src/`, `tests/`, a `conftest.py` that inserts the fixture root onto `sys.path` (needed because `python evaluation/run_reviews.py` doesn't put the repo root on `sys.path[0]` the way a script at the repo root, like `simulate_pr.py`, gets for free), and `meta.json` describing the injected issue, expected findings, and *why* the fixture's own test suite passes despite it — every fixture's tests pass by design, since the point is issues that slip past CI and need a reviewer to catch. `evaluation/manifest.json` indexes all 15 for the scripts below.

Three scripts, each independently runnable and built around Groq's tight daily budget rather than assuming one continuous run. All three accept `--results-dir <name>` (default `results`, relative to `evaluation/`), so a fresh pass (e.g. `results_v2`) can run without touching or being skipped by an existing completed pass.
- `evaluation/run_reviews.py` — feeds each fixture through the real `run_pr_review_agent`, checking `check_budget_ok()` before each one and stopping the moment it's exhausted. Each result is written to `<results-dir>/<id>.json` immediately (atomic temp-file + `os.replace`). A `status: "deferred"` result (a real Groq `RateLimitError` mid-review) is auto-retried on the next run; only `"completed"` is skipped. **Budget gate is per-UTC-date in Redis, not per-API-key** — swapping `GROQ_API_KEY` doesn't reset it; clear `usage:groq:*:{today}` manually after a genuine key swap. Groq's real per-org TPD limit is authoritative (confirmed: keys sharing the same org hit the identical 429 — check the org id in the 429 message, not just the key string). A 429's `Please try again in Xs` is a rolling-window refill, not a hard daily reset — often worth a short wait instead of a new key.
  - **Fake sha now includes `run_tag` (from `--results-dir`)**: `eval-{run_tag}-{fixture_id}`, not just `eval-{fixture_id}`. Reason: `app/agent.py`'s Redis checkpoints (no TTL) are keyed off this sha, so a fixed-per-fixture sha made *every* rerun of this script (different day, different `--results-dir`, a regression check) resume and keep growing the same checkpoint thread instead of starting fresh — silently multiplying prompt cost (confirmed: one fixture hit 47+ accumulated checkpoints, inflating a review from ~10-12k tokens to 50k+). Tying the tag to `--results-dir` keeps same-run deferred-retries working while any new `--results-dir` gets an isolated thread automatically, no manual Redis cleanup needed.
- `evaluation/evaluate_results.py` — fully offline (no Groq/Redis calls), scores whatever's in `<results-dir>/` via keyword-match against each fixture's `meta.json`. Cheap and recall-only; blind to false positives and to correct findings phrased differently than the keyword list — see `judge_results.py` for a deeper pass.
- `evaluation/judge_results.py` — Phase 5.5, an LLM-as-judge pass using Gemini (`gemini-3.5-flash` via the `google-genai` SDK — deliberately a different provider so grading never touches the Groq budget). Deliberately decoupled from `app/config.py`/`run_reviews.py`: reads `GEMINI_API_KEY` directly from the environment or `.env`, not through `Settings`, so it needs none of the Groq/GitHub secrets. Grades each stored review with a structured Pydantic verdict (`true_positive_caught`, `section_correct`, `severity_matched`, `false_positive_count`, `reasoning`) and reports Precision/Recall/an **F2 score** (recall weighted 4x over precision — a missed vulnerability costs more than an extra false positive) per category and overall. Same resumable/atomic-write pattern as `run_reviews.py`, plus: a Gemini 429 stops the run; a 5xx ("model currently experiencing high demand") retries up to 3x with linear backoff before giving up on that one fixture; a 400/401/403 (auth/permission failure, e.g. a bad key) stops the *whole run* immediately instead of grinding through the remaining fixtures against a key that will fail identically every time, and deliberately does not write a result file for that fixture — a written-but-failed judge record needs the same "was this ever actually resolved" check as `run_reviews.py`'s deferred-status handling, or it would look like "already graded" forever. Requests are throttled to at least 6 seconds apart to stay under Gemini's free-tier per-minute limit.

**Gemini API key gotcha, confirmed directly:** Google is retiring the legacy `AIza`-prefixed Gemini key format in favor of `AQ.`-prefixed "Auth keys" (started 2026-06-19, full `AIza` cutover 2026-09) — an `AQ.` key is legitimate and works fine with `genai.Client(api_key=...)`, not a sign of a broken or wrong key. The real gotcha: `_resolve_gemini_config()` checks the real environment before falling back to `.env` (same source-order convention as `pydantic-settings`) — a stale OS-level `GEMINI_API_KEY` environment variable set outside this project silently overrides every `.env` edit with no error, producing a confusing `400: API key not valid` even when the `.env` key is completely correct. If a key that looks right keeps failing, check for a pre-existing `GEMINI_API_KEY` in the shell/OS environment first (`[Environment]::GetEnvironmentVariable('GEMINI_API_KEY','User')` on Windows).
